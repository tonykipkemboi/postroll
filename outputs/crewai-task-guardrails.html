What Are Task Guardrails?

0:00

A task guardrail in CrewAI is a custom validation function you attach to a task. After the agent produces output, the guardrail runs and decides whether it meets your criteria — if not, CrewAI automatically retries, up to a configurable maximum.

The intuition: think of bumpers on a bowling lane. Without guardrails, an agent's output can drift into the gutter — too long, wrong format, missing fields. With a guardrail, you define the bumpers and the agent keeps trying until the output lands where you want it.

By the end of this tutorial you'll know how to:

  • Write a custom guardrail function
  • Attach it to a CrewAI Task
  • Configure max_retries so the agent self-corrects

from crewai import Agent, Task, LLM, Crew
from crewai_tools import SerperDevTool
from typing import Tuple, Any
from crewai.tasks.task_output import TaskOutput
import warnings
warnings.filterwarnings("ignore")

The key types are TaskOutput — the object your guardrail receives — and Tuple[bool, Any], which is what it must return. You'll also need API keys for your LLM provider and Serper (web search).

The Guardrail Function

3:03

A guardrail function always has the same signature: takes a TaskOutput, returns a Tuple[bool, Any]. True means pass; False triggers a retry with your message fed back to the agent.

def validate_blog_content(result: TaskOutput) -> Tuple[bool, Any]:
    try:
        word_count = len(result.raw.split())
        print(f"Word count: {word_count}")
        if word_count > 50:
            return (False, "Blog content exceeds 50 words")
        return (True, result.raw.strip())
    except Exception as e:
        return (False, f"Unexpected error: {e}")
When you return (False, "some message"), that message is sent back to the agent as feedback. The agent sees its prior output plus the failure reason and generates a new attempt — far more targeted than just prompting for shorter output upfront.

Building the Agent, Task & Crew

4:36 
llm = LLM(model="gpt-4o-mini", api_key=OPENAI_API_KEY)

blog_agent = Agent(
    role="Blog Writer",
    goal="Write blog post",
    backstory="An expert blog writer",
    tools=[SerperDevTool()],
    llm=llm,
    verbose=True
)

Notice {prompt} and {year} in the description — they're interpolated at runtime via crew.kickoff(inputs={...}).

blog_task = Task(
    description="Write a super DETAILED blog post about {prompt} for {year}",
    expected_output="A properly structured blog post under 200 words.",
    agent=blog_agent,
    markdown=True,
    guardrail=validate_blog_content,   # attach the function directly
    max_retries=3
)

guardrail takes the function directly (not a string). max_retries is the iteration cap. Then assemble the crew:

crew = Crew(agents=[blog_agent], tasks=[blog_task], verbose=True)

Adjusting the Criteria & Wrap-Up

7:39

To tighten to 50 words, update both the function threshold and expected_output:

# In validate_blog_content:
if word_count > 50:
    return (False, "Blog content exceeds 50 words")

# In blog_task:
expected_output="A properly structured blog post under 50 words."

The pattern generalises far beyond word count — validate JSON schema, required fields, external API checks, regex on structured output. The agent sees your failure message and self-corrects, making pipelines more deterministic without touching the underlying prompts or model.
